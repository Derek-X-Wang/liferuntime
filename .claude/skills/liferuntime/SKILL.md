# LifeRuntime

LifeRuntime is a deterministic, event-sourced runtime that models the user's strategic life. Inputs (Projects, Goals, Signals, Decisions, TimePulses) are appended to a log. Derived state (Project relevance, urgency, momentum, per-project Decision stance, Trajectories) is regenerated by replay. The same events always produce the same world.

Your job here is to **be the human's interface to that runtime** — translate natural-language strategic statements ("I had a great chat with X about TNT today", "I'm putting weight on TNT and shelving Side-X") into the right CLI invocations, ingest them, then summarize what changed.

## Mental model

```
Event in → World changes → Explanation out
```

- **Events are inputs only.** The user said it happened (or an external signal arrived). Events go in the log. (ADR-0003)
- **Systems derive state per event.** Matching, decay, Decision stance — all run on every event, not at advance time. (ADR-0006)
- **`advance` is a reporter, not a trigger.** Calling `advance` doesn't *cause* anything; it prints what's changed since the last `advance`. (ADR-0002)
- **Time is event-log time, not wall-clock.** Decay measures days by Events in the log, not by your phone clock. Use `pulse` to push event-log time forward without an underlying world event (ADR-0004).
- **Inspection is always live.** `inspect`, `status`, `decision list`, `explain` all see the fully-derived current state whether or not you've called `advance`.

The vocabulary in [`docs/CONTEXT.md`](../../../docs/CONTEXT.md) is canonical — use the exact terms (Resonance, Trajectory, Cause, Decision, stance, ChangeRecord). If the user uses a fuzzy term, propose the canonical one ("you're saying 'commit' — do you mean a `decide` event with `--chose`?").

## Boot check

Every invocation:

1. **Confirm the binary**. Prefer `liferuntime` on `PATH` (user can `cargo install --path apps/cli` from the repo). Otherwise use `<repo>/target/release/liferuntime`. If neither, tell the user to build or install before continuing — don't fall back to `cargo run` per-command (slow + spammy).
2. **Confirm the world dir**. Default: `~/.liferuntime`. **Always pass `--dir ~/.liferuntime` explicitly** — the CLI default is `.liferuntime` (cwd), which is wrong for the user's strategic world (it should be one global lifeworld, not per-directory). If the dir doesn't exist, run `liferuntime --dir ~/.liferuntime init` once before anything else.

If a different `--dir` is justified (scenario testing, a separate experimental world), say so explicitly and ask the user to confirm.

## Verb cheat sheet

All commands take `--dir ~/.liferuntime` (omitted below for brevity — always include it).

### Setup

```
liferuntime init                     # create empty world (one-time)
```

### Entities

```
liferuntime project add <id> --name "<name>" [--tags a,b,c]
liferuntime project edit <id> [--name ...] [--tags ...] [--depends-on x,y]
liferuntime project archive <id> [--reason "..."]
liferuntime project complete <id> [--note "..."]
liferuntime project reactivate <id>

liferuntime goal add <id> --name "<name>" [--tags a,b] [--importance 0.0..1.0]
liferuntime goal edit <id> [--name ...] [--tags ...] [--importance ...]
liferuntime goal achieve <id> [--note "..."]
liferuntime goal abandon <id> [--reason "..."]
liferuntime goal reactivate <id>
```

`id` is the user-chosen handle (e.g. `tnt`, `side-x`, `health`). Lowercase + hyphen. Stable forever — referenced by every later event.

### Signals

```
liferuntime signal add --source "<where>" --summary "<one line>" \
    [--tags a,b] [--confidence 0.0..1.0] [--idempotency-key K]

liferuntime signal analyze "<free text>" [--tags a,b] [--commit]
```

`signal analyze` runs the FakeAgent stub today — it proposes structured signals from free text. Without `--commit` it prints proposals; with `--commit` it ingests them. The runtime never auto-ingests proposals.

### Time

```
liferuntime pulse                    # advance event-log time to wall-clock now
liferuntime pulse --at 2026-05-21T09:00:00Z   # explicit timestamp
liferuntime pulse --idempotency-key cron-2026-05-21
```

### Strategy

```
liferuntime decide --chose <id> \
    [--over a,b]                     # narrative rivals; no mechanic
    [--dampen c,d]                   # explicit suppression set
    [--because "rationale"]
    [--decided-at TS]

liferuntime decision list            # active per-project stances
liferuntime decision revoke <decision-id>
```

`--over` and `--dampen` are **separate**. `--over` is history-only — narrative rivals named in `explain`. `--dampen` is the mechanic — listed projects have their resonance deltas × 0.3 and Goal amplification suppressed. Most invocations want **both** the rival named *and* mechanically dampened: pass the same ids to both flags.

### Reading

```
liferuntime status [--window 5]      # active projects ranked by relevance, with ↑↓→ trajectory
liferuntime inspect project <id>     # current strategic_relevance (raw + visible), urgency, status, stance, tags, depends_on
liferuntime explain latest           # most-recent-Advance window
liferuntime explain project <id>     # default: latest window for this project
liferuntime explain project <id> --all   # full causal history — every ChangeRecord that ever touched this project
liferuntime replay                   # smoke check: number of stored events
liferuntime advance                  # mark the cursor; reports what's new since last advance
```

## Tag conventions

Tags are how Signals match Projects (Resonance) and Goals amplify (Goal-amp). Canonicalization is automatic in v1: lowercase, trim, collapse spaces/underscores to hyphens. `AI Voice`, `ai_voice`, and `ai-voice` all match. Synonyms (`AI` vs `artificial-intelligence`) do **not** — the user must pick one canonical form per concept.

### Starter pillars

Use these as recurring tags so Goals can amplify whole life-areas:

- `health`
- `financial-freedom`
- `moxiang`
- `family`

### Tech / project tags

Grow organically. When the user mentions a new technology, project area, or external entity, propose a canonical tag (lowercase, hyphenated, singular if possible) and reuse it forever. Examples: `ai-voice`, `speech-research`, `bite`, `tnt`. **Never invent a synonym** for an existing tag — if `ai-voice` is canonical, don't later use `voice-ai`.

When you (the skill) help compose a Signal or Project, list the proposed tags in the response and explicitly call out any new ones so the user can correct.

## Workflows

### W1. First-time setup

1. `liferuntime --dir ~/.liferuntime init`
2. Add the user's top 3-5 active Projects via `project add`. Use real ids the user already uses (lowercase, hyphen-separated).
3. Add Goals representing the strategic horizons the user cares about. Use the starter pillars as Goal tags so Signals matching pillar-tagged Projects amplify naturally.
4. Optionally seed a few existing Signals from the past week or two so the world isn't dead-empty.

### W2. Ingest a signal (most common)

User says: *"I had a great call with the speech-research lab today about voice agents."*

1. Identify which Project(s) this is most relevant to (`tnt`? `speech-research-project`?). Confirm with the user if unsure — don't guess.
2. Propose tags: `ai-voice`, `speech-research`. Confirm new tags with the user.
3. Run:
   ```
   liferuntime signal add --source "conversation" \
       --summary "Call with speech-research lab — voice agents direction" \
       --tags ai-voice,speech-research --confidence 0.8
   ```
4. Report the resulting ChangeRecord(s) — which Projects moved, by how much, what Causes (Goal amp, Decision stance).

For batch ingest from a paragraph of notes: try `signal analyze "<paragraph>"` first (no `--commit`), show the proposed signals, then re-run with `--commit` after the user approves.

### W3. Pulse forward (time passing)

The runtime moves forward only when the user (or a cron) says it does. Helpful when no signals have arrived for days/weeks but Decay should still erode stale relevance.

```
liferuntime pulse                   # right now
liferuntime pulse --at 2026-05-20T22:00:00Z  # last night, e.g. weekly catch-up
```

For nightly cron use `--idempotency-key cron-<date>` so a retried run doesn't double-pulse.

### W4. "What's pulling on my attention?"

```
liferuntime status
```

Reads active Projects ranked by visible relevance with Trajectory arrows. Use the `--window` flag to widen or narrow the slope-over-Advances window (default 5).

When the user asks loose questions like "what should I focus on" / "what's hot" / "give me a strategic check-in", this is the first thing to run. Pair with `decision list` if Decisions are active so the user sees what they've committed to.

### W5. Trace the cause of a state shift

User asks: *"Why is TNT at 0.78?"* or *"How did Side-X get cold?"*

```
liferuntime explain project tnt --all
```

Returns every ChangeRecord touching that Project, in chronological order, with Causes (matching Signal, Goal amp, Decision boost / dampen, Decay). Use this whenever the user expresses surprise at a state value — never speculate; let `explain --all` answer.

### W6. Commit a Decision

User says: *"I'm putting serious weight on TNT this quarter and want Side-X to stop pulling on me."*

1. Confirm: chose Project = `tnt`. Rivals = `side-x`. Mechanic = dampen Side-X too (the user wants it to stop pulling).
2. Run:
   ```
   liferuntime decide --chose tnt --over side-x --dampen side-x \
       --because "Q3 focus on TNT; Side-X explicit suppression"
   ```
3. Surface the resulting effect: `tnt` gets a +0.15 boost decaying at ≈0.999/day; `side-x` resonance deltas scaled ×0.3, Goal amp suppressed.
4. Show `decision list` so the user sees what's currently in force.

If the user later says *"actually I want to undo that decision"* without naming a replacement:
```
liferuntime decision revoke <decision-id>
```
The id is the event id of the `DecisionRecorded` — copy it from `decision list` output.

### W7. Project lifecycle (archive / complete)

- **Archive**: soft-shelve; reactivatable. *"I'm pausing Side-X until 2027."*
- **Complete**: done; historical record. *"Shipped TNT v1; marking complete."*
- **Reactivate**: bring an Archived or Completed Project back to Active. Use sparingly — completion that gets reversed should be rare and noteworthy.

Goals have the symmetric verbs: `achieve` / `abandon` / `reactivate`. Use the value-charged verb that matches the user's framing: `achieve` for "I reached it"; `abandon` for "I gave up on it."

## Common pitfalls

- **Forgetting `--dir`**: the CLI defaults to cwd. Always pass `--dir ~/.liferuntime` (or whatever the user's world dir is).
- **Inventing tag synonyms**: if a Signal uses `voice-ai` when the canonical tag is `ai-voice`, it *will* match (auto-canonicalization handles `voice-ai` and `ai-voice` similarly only for hyphen/underscore/space normalization — NOT for word reordering). Treat unfamiliar tags as a prompt to check existing tags before introducing.
- **Calling `advance` to "make it run"**: `advance` is a reporter. Systems already ran when each event was ingested. Use `advance` when the user wants the "what's new since last check-in" view; don't call it speculatively.
- **Referencing unknown ids**: `decide --over <id>`, `project edit --depends-on <id>` etc. all validate that the target Project exists at ingest. If you propose an invocation that references a Project the user hasn't created, create it first or omit it.
- **Skipping `--idempotency-key` on cron-driven `pulse` / `signal add`**: a retry doubles the event. Use a stable key for any automated invocation.
- **Reading `strategic_relevance` without the raw/visible split**: `inspect` shows both `strategic_relevance_raw` and `strategic_relevance_visible`. The visible value includes the active Decision boost (if any). Cite both when explaining state — the gap *is* the Decision's contribution.

## Provenance discipline

When the user asks "why did X move?", **never speculate** about causes from a single state read. Always run `explain project <id> --all` (or `explain latest`) and quote the recorded Causes. The runtime's value depends on this trust surface — every Change traces to recorded Events. If `explain` can't justify a state value, that's a runtime bug worth surfacing, not something to paper over.
